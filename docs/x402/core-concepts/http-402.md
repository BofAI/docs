# HTTP 402

## What is HTTP 402?

[HTTP 402](https://datatracker.ietf.org/doc/html/rfc7231#section-6.5.2) is a standard but historically underutilized HTTP response status code intended to indicate that “payment is required to access the requested resource.”

In the x402 protocol, we reactivate this status code to:

- **Inform** the client (buyer or agent) that access is restricted and payment is required.
- **Convey** detailed payment requirements, including amount, token type, and destination address.
- **Provide** the necessary metadata for constructing a programmatic payment (used to build the payload to be signed).

## Why Use HTTP 402?

The core purpose of adopting HTTP 402 in x402 is to enable a **frictionless, API-native** payment experience for web resources, especially suitable for:

- **Machine-to-Machine (M2M) Payments**: Such as autonomous transactions between AI Agents.
- **Pay-per-Use Models**: Such as API metering or paywalled content.
- **Micropayments**: Small payments without account registration or traditional fiat payment rails.
- **Global Payments**: Settlement via blockchain-based stablecoins (e.g., USDT), eliminating geographic restrictions.

By reusing the standard 402 status code, x402 maintains strong compatibility with native web protocols and can be easily integrated into any HTTP-based service architecture.

## Payment Headers

x402 defines a set of standardized HTTP headers for payment communication:

- **`PAYMENT-REQUIRED`**: Contains the **payment requirements** generated by the server (Base64-encoded). This header is returned with a `402` response and specifies detailed payment parameters.
- **`PAYMENT-SIGNATURE`**: Contains the **payment payload** generated by the client (Base64-encoded). After receiving the 402 response, the client retries the request with this header as cryptographic proof of authorized payment.
- **`PAYMENT-RESPONSE`**: Contains the **settlement response** returned by the server (Base64-encoded). This header accompanies a successful response (e.g., `200 OK`) and includes settlement proof such as the transaction hash.

> **Technical Note**: The values of these headers must be **valid Base64-encoded JSON strings**. This encoding ensures compatibility across different HTTP implementations and prevents parsing issues caused by special characters in JSON payloads.

**Key Fields:**

| Field               | Description                                                                                          |
| ------------------- | ---------------------------------------------------------------------------------------------------- |
| `x402Version`       | Protocol version (currently 2)                                                                       |
| `error`             | Human-readable error message                                                                         |
| `resource`          | Information about the requested resource                                                             |
| `accepts`           | Array of accepted payment options                                                                    |
| `scheme`            | Payment scheme (`exact`)                                                                             |
| `network`           | Network identifier (`tron:nile`, `tron:mainnet`, `eip155:56`, `eip155:97`)                           |
| `amount`            | Payment amount in the smallest unit (e.g., 100 = 0.0001 USDT)                                        |
| `asset`             | TRC-20/BEP-20 token contract address                                                                 |
| `payTo`             | Seller's wallet address                                                                              |
| `maxTimeoutSeconds` | Maximum validity duration of the payment                                                             |
| `extra.fee`         | Facilitator fee information (includes `facilitatorId`, `feeTo`, `feeAmount`, `caller`)               |
| `extensions`        | Additional context for the payment scheme (e.g., `paymentPermitContext` with nonce, validity window) |

## Summary

HTTP 402 is the foundation of the x402 protocol. It enables servers to integrate payment logic directly at the HTTP response layer. Through this standard status code, the protocol achieves:

- **Payment Signaling**: Clearly marking resources as “pay-to-access.”
- **Parameter Transmission**: Accurately conveying key metadata such as amount, token contract, and recipient address.
- **Seamless Integration**: Full compatibility with existing standard HTTP workflows (stateless and RESTful).
- **Programmatic Settlement**: Fully automated, on-chain payment execution without manual intervention.
