# 信誉系统

## ERC-8004 上的信誉机制是如何工作的？
ERC-8004 本身并非一套信誉系统或算法：它是一个在公共注册表中存储信誉数据的标准（因此它成为了一种公共物品），以便任何构建平台、系统或信誉算法的人都可以使用它。

## 反馈的结构是怎样的？
反馈由任何 `clientAddress`（客户端地址）签名，并指向记录该反馈的链上的某个 Agent 的 `agentId`。同一个 `clientAddress` 可以多次向同一个 Agent 提供反馈，因此链上的反馈标识为：`agentId:clientAddress:feedbackIndex`。

一个反馈条目包含：
- `value` (int128) - 反馈评分（必填）
- `valueDecimals` (uint8) - 数值的小数精度
- `tag1` 和 `tag2` (string) - 用于增加反馈维度的标签（可选）
- `endpoint` (string) - 反馈所指的具体服务端点/路由（可选）
- `feedbackURI` (string) - 指向包含更多反馈详情的链下文件的链接（可选）
- `feedbackHash` (bytes32) - 用于完整性验证的链下内容哈希值（可选）

如果你需要存储更多信息，可以将其放在由 `feedbackURI` 链接的 JSON 文件中，并通过 `feedbackHash` 验证其完整性。

## 信誉有哪些用例？
`tag1` 字段定义了 `value` 衡量的内容。常见示例包括：

- `starred` - 质量评分
- `reachable` - 端点可达性
- `ownerVerified` - 端点由 Agent 所有者拥有
- `uptime` - 端点在线率 %
- `successRate` - 端点成功率 %
- `responseTime` - 响应时间（毫秒）
- `blocktimeFreshness` - 平均区块延迟（块数）
- `revenues` - 累计收入
- `tradingYield` - 收益率（配合 `tag2` 指定周期：日、周、月、年）

## 谁来聚合信誉数据？
任何人——数据是公开的。不仅仅是 Agents 和平台：我们预计（就像传统金融中已经发生的那样）会出现专门的公司，利用 ERC-8004 提供的数据对 Agents 和服务进行信任排名。

## 如何防止垃圾信息攻击 (Spam Attacks)？
开放系统中的信誉是一个开放且非常复杂的问题，没有单一的万能解决方案。ERC-8004 没有强制执行特定的身份或质押要求——这只适用于部分用例和利益相关者——而是做出了一个中立的选择：**作为信息通道**，将反垃圾过滤工作委托给平台实现者和算法聚合方。

在短期内，我们预计开发者会通过过滤他们信任的组织/评分者的 `clientAddresses` 来进行筛选。从长远来看，可能会发展出更先进的方法——例如，根据评分者自身收到的反馈自动计算评分者的信誉，从而评估其发出的反馈权重（递归信誉）。
